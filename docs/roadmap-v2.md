# Стратегия C: Трансформация Smailo в Low-Code платформу

## Контекст

Smailo начинался как эксперимент "можно ли через чат с AI создавать приложения без кода". Текущая архитектура — это конструктор данных: 21 PrimeVue-компонент, flat key-value хранилище, 5 типов cron-задач. Это хорошо работает для дашбордов и простых трекеров, но не позволяет строить интерактивные приложения с логикой, несколькими страницами, реляционными данными или мульти-пользовательским доступом.

Решение: поэтапная трансформация в low-code платформу. Каждый этап — отдельный план, самодостаточный и тестируемый.

---

## Общие принципы

- **Эволюция, не революция** — не переписываем всё сразу; каждый этап добавляет слой поверх работающей системы
- **Обратная совместимость** — существующие приложения (flat KV) продолжают работать на каждом этапе
- **Дорожная карта живая** — этот документ будет обновляться по мере реализации; порядок и содержание этапов могут меняться
- **AI-first** — каждая новая возможность должна быть доступна через чат с AI, не только через UI
- **Тесты обязательны** — каждый этап включает unit-тесты (vitest) для серверной логики; все тесты должны проходить перед завершением этапа; новый код без тестов не считается завершённым

## Дорожная карта (8 этапов)

### Этап 1: Пользовательские таблицы данных ✅
**Статус:** Реализован
**Зависимости:** нет (фундамент для всего остального)

**Что сделано:**
- Новые таблицы в БД: `user_tables` (схема таблиц пользователя) и `user_rows` (данные)
- API: CRUD для таблиц и строк (`/api/app/:hash/tables`, `/api/app/:hash/tables/:tableId/rows`)
- Типы колонок: text, number, date, boolean, select (enum)
- Валидация типов при записи
- Совместимость: старый `app_data` (KV) продолжает работать параллельно

**Тесты:** нет (vitest ещё не был настроен)

---

### Этап 2: Привязка компонентов к таблицам ✅
**Статус:** Реализован
**Зависимости:** Этап 1

**Что сделано:**
- Новое поле `dataSource: { type: "table", tableId }` в конфиге компонентов (альтернатива `dataKey`)
- `DataTable` отображает строки из пользовательской таблицы с авто-генерацией колонок из схемы
- `Form` записывает строки в пользовательскую таблицу с авто-генерацией полей из схемы (text, number, date, boolean, select)
- `Chart` строит графики по данным из таблиц (первая колонка = метки, числовые колонки = датасеты)
- `CardList` показывает строки из таблицы как карточки с поддержкой удаления
- Данные таблиц кэшируются в Pinia-сторе и обновляются при записи/удалении
- Полная обратная совместимость: `dataKey` продолжает работать для всех компонентов

**Тесты:** vitest настроен; 112 тестов (валидация компонентов, таблиц, строк, промптов, chartData)

---

### Этап 3: Формулы и вычисляемые поля ✅
**Статус:** Реализован
**Зависимости:** Этап 1

**Что сделано:**
- Безопасный парсер формул (рекурсивный спуск, без eval): арифметика, сравнения, логика, строковые операции, IF/THEN
- Вычисляемые колонки в таблицах: `{ name: "total", type: "formula", formula: "price * quantity" }` — вычисляются per-row на сервере при чтении
- Вычисляемые поля в конфиге компонентов: `computedValue: "= SUM(expenses.amount)"` — агрегаты по таблицам, вычисляются на сервере
- Агрегатные функции: SUM, AVG, COUNT, MIN, MAX по колонке таблицы
- Встроенные функции: IF, ABS, ROUND, FLOOR, CEIL, MIN, MAX, UPPER, LOWER, CONCAT, LEN, TRIM, NOW
- Существующие compute/aggregate_data cron jobs работают без изменений (обратная совместимость)
- AI-промпты обновлены с документацией по формулам

**Тесты:** unit-тесты для токенизатора, парсера, вычислителя, встроенных функций, агрегатов, валидации формульных колонок, computedValue, AI-промптов

---

### Этап 4: Условная логика и видимость ✅
**Статус:** Реализован
**Зависимости:** Этап 3 (нужен парсер выражений)

**Что сделано:**
- `showIf: string` в конфиге компонента: `showIf: "status == 'active'"` или `showIf: "balance < 0"` — компонент скрыт когда выражение falsy
- `styleIf: [{ condition: string, class: string }]` — условные CSS-классы; предустановленные: `warning` (жёлтый), `critical` (красный), `success` (зелёный), `muted` (серый), `highlight` (синяя подсветка)
- `ConditionalGroup` — компонент-контейнер, показывает/скрывает группу вложенных компонентов по условию; max 1 уровень вложенности
- Парсер формул портирован на клиент (`client/src/utils/formula/`); вычисление showIf/styleIf происходит реактивно по изменению appData/tableData
- Серверная валидация showIf/styleIf выражений через `parseFormula()` при сохранении; невалидные выражения удаляются
- `AppConditionalGroup.vue` — клиентский компонент для рендеринга ConditionalGroup
- AI-промпты обновлены с документацией по showIf/styleIf/ConditionalGroup

**Тесты:** unit-тесты для парсера формул (клиент), evaluateShowIf, evaluateStyleIf, buildFormulaContext, серверной валидации showIf/styleIf, ConditionalGroup валидации, AI-промптов

---

### Этап 5: Многостраничность и навигация ✅
**Статус:** Реализован
**Зависимости:** Этапы 1-2

**Что сделано:**
- Массив `pages` в конфиге приложения: `[{ id, title, icon?, uiComponents[] }]` (max 10 страниц)
- Навигация через PrimeVue Tabs над контентом, URL: `/:userId/:hash/:pageId`
- AI-ответ расширен полем `pagesUpdate?: Page[]` (заменяет весь массив pages)
- Глобальные индексы computedValues пересчитываются в локальные индексы текущей страницы
- Обратная совместимость: приложения без `pages` = одна страница (как раньше)

**Тесты:** unit-тесты для validatePages, getGlobalComponents, pages getter, localComputedValues offset, router, AI-промптов

---

### Этап 5.5: Фильтрация строк в dataSource
**Статус:** ✅ Реализован
**Зависимости:** Этап 2 (привязка к таблицам)

**Что сделано:**
- Поле `filter` в `dataSource`: `{ type: "table", tableId: N, filter: { column, operator?, value } }` или массив условий (AND)
- Операторы: `eq` (по умолчанию), `ne`, `lt`, `lte`, `gt`, `gte`, `contains` (регистронезависимый)
- Серверная фильтрация в памяти после `evaluateFormulaColumns()` (без SQL JSON операций)
- Клиентский кэш учитывает фильтр в ключе: `tableId:JSON(filter)`
- Поддержка в компонентах DataTable, CardList, Chart; Form не поддерживает (фильтр удаляется при валидации)
- AI-промпты обновлены с примерами фильтрации
- Обратная совместимость: без `filter` — поведение не меняется

**Контекст:** Потребность обнаружена при тестировании Этапа 5 — многостраничные приложения с фильтрацией по колонке (например, задачи по приоритету) не работают без серверного WHERE

**Тесты:** unit-тесты для валидации filter в dataSource, серверной фильтрации строк (все операторы, null значения, AND логика), клиентского кэширования, AI-промптов

---

### Этап 6: Event-система и цепочки действий ✅
**Статус:** Реализован
**Зависимости:** Этапы 1-4

**Что сделано:**
- Поле `actions: ActionStep[]` в конфиге Button/InputText/Form — упорядоченный список шагов, выполняемых последовательно при взаимодействии пользователя (max 5 шагов в цепочке)
- 5 типов действий: `writeData` (запись в appData с mode: append/increment/delete-item), `navigateTo` (навигация по страницам), `toggleVisibility` (переключение boolean для showIf), `runFormula` (вычисление формулы на клиенте), `fetchUrl` (прокси-запрос HTTPS через сервер с SSRF-защитой)
- Серверная валидация цепочек: max 5 шагов, per-type проверки полей, невалидные шаги удаляются
- Миграция: legacy `action: { key, value }` автоматически конвертируется в `actions[0]` типа `writeData` при сохранении; клиент поддерживает оба формата
- Клиентский исполнитель `executeActions()` — последовательное выполнение шагов с обработкой ошибок (auth ошибка → цепочка прерывается, прочие → продолжается)
- Общий модуль `fetchProxy.ts` — SSRF-безопасный HTTP fetch (проверка приватных IP, DNS rebinding, redirect blocking, 1MB/10s лимиты) + `extractDataPath()` для dot-notation извлечения данных
- Эндпоинт `POST /api/app/:hash/actions/fetch-url` — серверный прокси для fetchUrl действий с rate limiting (30 req/min)
- AI-промпты обновлены с документацией по actions и примерами всех 5 типов

**Тесты:** unit-тесты для валидации цепочек (все типы, max 5, миграция action→actions), fetchProxy (isPrivateHost, fetchSafe, extractDataPath), клиентского исполнителя (все типы, цепочки, ошибки, legacy fallback), AI-промптов

**Статус:** Запланирован

## Порядок реализации и зависимости

```
Этап 1: Таблицы ─────────┬──→ Этап 2: Привязка данных ──→ Этап 5: Страницы
                          │
                          ├──→ Этап 3: Формулы ──→ Этап 4: Условия
                          │
                          └──→ Этап 8: Мульти-юзер

Этапы 1-6 ──────────────────→ Этап 7: Визуальный редактор

Этап 6 (Events) зависит от Этапов 1-4
```

Этапы 3 и 5 можно делать параллельно. Этап 8 можно делать после Этапа 2 в любой момент.

## Что НЕ входит в дорожную карту (осознанно отложено)

| Что | Почему отложено |
|---|---|
| Sandbox для пользовательского кода | Серьёзная проблема безопасности; формулы покрывают 80% сценариев |
| Кастомный CSS / темы | Низкий приоритет; PrimeVue Aura достаточно хорош |
| Push-уведомления | Требует инфраструктуру (FCM/APNs), не связано с ядром платформы |
| Маркетплейс шаблонов | Имеет смысл только после набора базы пользователей |
| Мобильное приложение | Веб работает на мобильных; нативное приложение — отдельный проект |
