# Стратегия C: Трансформация Smailo в Low-Code платформу

## Контекст

Smailo начинался как эксперимент "можно ли через чат с AI создавать приложения без кода". Текущая архитектура — это конструктор данных: 21 PrimeVue-компонент, flat key-value хранилище, 5 типов cron-задач. Это хорошо работает для дашбордов и простых трекеров, но не позволяет строить интерактивные приложения с логикой, несколькими страницами, реляционными данными или мульти-пользовательским доступом.

Решение: поэтапная трансформация в low-code платформу. Каждый этап — отдельный план, самодостаточный и тестируемый.

---

## Общие принципы

- **Эволюция, не революция** — не переписываем всё сразу; каждый этап добавляет слой поверх работающей системы
- **Обратная совместимость** — существующие приложения (flat KV) продолжают работать на каждом этапе
- **Дорожная карта живая** — этот документ будет обновляться по мере реализации; порядок и содержание этапов могут меняться
- **AI-first** — каждая новая возможность должна быть доступна через чат с AI, не только через UI

---

## Дорожная карта (8 этапов)

### Этап 1: Пользовательские таблицы данных ✅
**Статус:** Реализован
**Зависимости:** нет (фундамент для всего остального)

**Что сделано:**
- Новые таблицы в БД: `user_tables` (схема таблиц пользователя) и `user_rows` (данные)
- API: CRUD для таблиц и строк (`/api/app/:hash/tables`, `/api/app/:hash/tables/:tableId/rows`)
- Типы колонок: text, number, date, boolean, select (enum)
- Валидация типов при записи
- Совместимость: старый `app_data` (KV) продолжает работать параллельно

---

### Этап 2: Привязка компонентов к таблицам
**Статус:** Запланирован
**Зависимости:** Этап 1

**Что делаем:**
- Новый тип `dataSource` в конфиге компонентов: `{ type: "table", tableId, filter?, sort?, limit? }`
- `DataTable` умеет отображать строки из пользовательской таблицы (не только из KV)
- `Form` умеет записывать строки в пользовательскую таблицу (не только в KV)
- `Chart` умеет строить графики по данным из таблиц
- `CardList` умеет показывать строки из таблицы как карточки

---

### Этап 3: Формулы и вычисляемые поля
**Статус:** Запланирован
**Зависимости:** Этап 1

**Что делаем:**
- Парсер формул (безопасный, без eval): арифметика, сравнения, строковые операции, IF/THEN
- Вычисляемые колонки в таблицах: `{ name: "total", type: "formula", formula: "price * quantity" }`
- Вычисляемые поля в конфиге компонентов: `computedValue: "= SUM(expenses.amount)"`
- Агрегатные функции: SUM, AVG, COUNT, MIN, MAX по колонке таблицы
- Замена текущего `compute` cron job на формулы (обратная совместимость через обёртку)

---

### Этап 4: Условная логика и видимость
**Статус:** Запланирован
**Зависимости:** Этап 3 (нужен парсер выражений)

**Что делаем:**
- `showIf` в конфиге компонента: `showIf: "status == 'active'"` или `showIf: "balance < 0"`
- Использует тот же парсер формул из Этапа 3
- `ConditionalGroup` — компонент-контейнер, показывает/скрывает группу вложенных компонентов
- Условные стили: `styleIf: [{ condition: "value > 100", class: "warning" }]`

---

### Этап 5: Многостраничность и навигация
**Статус:** Запланирован
**Зависимости:** Этапы 1-2

**Что делаем:**
- Массив `pages` в конфиге приложения: `[{ id, title, icon?, components[] }]`
- Навигация между страницами (табы или сайдбар)
- Каждая страница — свой набор компонентов (но общие данные)
- URL: `/:userId/:hash/:pageId`
- Обратная совместимость: приложения без `pages` = одна страница (как сейчас)

---

### Этап 6: Event-система и цепочки действий
**Статус:** Запланирован
**Зависимости:** Этапы 1-4

**Что делаем:**
- Действия (actions) в компонентах: `onClick`, `onChange`, `onSubmit`
- Типы действий: `writeData`, `navigateTo`, `toggleVisibility`, `runFormula`, `fetchUrl`
- Цепочки: `actions: [{ type: "writeData", ... }, { type: "runFormula", ... }]`
- Серверная валидация цепочек (нет циклов, max 5 действий в цепочке)

---

### Этап 7: Визуальный редактор (drag-and-drop)
**Статус:** Запланирован
**Зависимости:** Этапы 1-6

**Что делаем:**
- Режим редактирования приложения (переключатель "чат / редактор")
- Drag-and-drop компонентов из палитры на холст
- Property editor — панель настройки выбранного компонента
- Привязка данных через UI (выбор таблицы, колонки)
- Layout grid — размещение компонентов в сетке (не только линейный список)
- Превью в реальном времени

---

### Этап 8: Мульти-пользовательский доступ
**Статус:** Запланирован
**Зависимости:** Этапы 1-2

**Что делаем:**
- Роли внутри приложения: `owner`, `editor`, `viewer`
- Приглашение пользователей по ссылке
- Права доступа: кто может читать/писать какие таблицы
- Row-level security: пользователь видит только свои строки (опционально)
- Таблица `app_members`: `{ appId, userId, role }`

## Порядок реализации и зависимости

```
Этап 1: Таблицы ─────────┬──→ Этап 2: Привязка данных ──→ Этап 5: Страницы
                          │
                          ├──→ Этап 3: Формулы ──→ Этап 4: Условия
                          │
                          └──→ Этап 8: Мульти-юзер

Этапы 1-6 ──────────────────→ Этап 7: Визуальный редактор

Этап 6 (Events) зависит от Этапов 1-4
```

Этапы 3 и 5 можно делать параллельно. Этап 8 можно делать после Этапа 2 в любой момент.

## Что НЕ входит в дорожную карту (осознанно отложено)

| Что | Почему отложено |
|---|---|
| Sandbox для пользовательского кода | Серьёзная проблема безопасности; формулы покрывают 80% сценариев |
| Кастомный CSS / темы | Низкий приоритет; PrimeVue Aura достаточно хорош |
| Push-уведомления | Требует инфраструктуру (FCM/APNs), не связано с ядром платформы |
| Маркетплейс шаблонов | Имеет смысл только после набора базы пользователей |
| Мобильное приложение | Веб работает на мобильных; нативное приложение — отдельный проект |

## Что переживает из текущего кода

| Модуль | Статус |
|---|---|
| Auth (bcrypt + JWT, creationToken) | Переживает, расширяется ролями |
| User management (users table, API) | Переживает без изменений |
| Axios interceptor + rate limiting | Переживает |
| PrimeVue компоненты | Переживают, добавляются новые |
| Smailo.vue (маскот + настроение) | Переживает |
| CronManager (инфраструктура) | Переживает, action types расширяются |
| Chat UI (InputBar, SpeechRecognition) | Переживает |
| Two-column layout (app + assistant) | Переживает, становится одним из режимов |
